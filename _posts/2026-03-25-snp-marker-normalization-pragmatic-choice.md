> 유전자검사 서비스의 가중치 테이블에서 SNP 데이터 중복이 보였다. 정규화 본능이 발동해서 마스터 테이블 분리를 설계했다. 그런데 한 발짝 물러서 보니, 실제 문제는 테이블 구조가 아니라 수정 쿼리의 WHERE 절이었다.

---

## 빠른 요약

- 문제: 같은 SNP(rs6152)의 OR/Beta 값이 패키지마다 복제되어 있어서, 논문 갱신 시 5곳을 각각 수정해야 했다.
- 검토한 방법: `snp_marker` 마스터 테이블을 분리하고 JOIN으로 쓰는 정규화 설계.
- 결론: 테이블을 분리하지 않고, 기존 테이블에 `rs_number + gene` 기준 일괄 수정 API를 추가하는 것으로 충분했다.
- 교훈: 중복 데이터가 보인다고 바로 정규화부터 하면 안 된다. 진짜 문제가 "저장 구조"인지 "수정 방식"인지를 먼저 구분해야 한다.

## 1. 상황

> 이 글은 [유전자검사 결과 저장: JSON → EAV 마이그레이션 회고](/posts/eav-vs-jsonb-gene-data-retrospective/)와 [유전자검사 서비스에 벤더사가 붙기 시작했다 — 멀티테넌시를 고민할 시점](/posts/gene-test-vendor-multi-tenancy-architecture/)의 연장이다. EAV로 결과 저장을 유연하게 바꾸고, B2B 벤더 구조를 올린 뒤, 이번에는 가중치 데이터 관리 쪽에서 문제가 보이기 시작했다.

유전자검사 서비스에서 점수를 계산하려면 SNP별 가중치 데이터가 필요하다.

```
DNA 원시 데이터에서 SNP 변이를 읽고
→ 해당 SNP의 Beta(가중치)를 곱해서
→ 항목별 점수를 합산하고
→ 등급 기준과 비교해서 "주의" / "보통" / "안심"을 판정한다
```

이 가중치 데이터를 담고 있는 테이블이 `weighted_info`다.

```
weighted_info
┌─────────┬──────────────┬──────┬────────┬──────┬──────┬────────┐
│ plan_id │ probeset_id  │ gene │ rs_num │ risk │ OR   │ Beta   │
│─────────│──────────────│──────│────────│──────│──────│────────│
│ 1 (A사) │ AX-11161415  │ AR   │ rs6152 │ G    │ 1.35 │ 0.3001 │
│ 2 (B사) │ AX-11161415  │ AR   │ rs6152 │ G    │ 1.35 │ 0.3001 │
│ 3 (C사) │ AX-11161415  │ AR   │ rs6152 │ G    │ 1.35 │ 0.3001 │
│ 4 (D사) │ AX-11161415  │ AR   │ rs6152 │ G    │ 1.35 │ 0.3001 │
│ 5 (자사) │ AX-11161415  │ AR   │ rs6152 │ G    │ 1.35 │ 0.3001 │
└─────────┴──────────────┴──────┴────────┴──────┴──────┴────────┘
```

이걸 보면 눈에 바로 들어온다. **5행이 전부 같은 데이터다.** `plan_id`만 다르고 나머지는 동일하다.

여기서 문제의식이 시작됐다.

## 2. 문제: 논문이 갱신되면 5곳을 각각 고쳐야 한다

유전자 가중치 데이터는 논문 기반이다. GWAS(Genome-Wide Association Study) 연구 결과가 업데이트되면 OR/Beta 값이 바뀐다.

예를 들어 rs6152의 OR이 1.35에서 1.42로 갱신됐다고 하자.

지금 구조에서는:

```
1. A사 패키지 상세 → 마커 목록에서 rs6152 찾아서 수정
2. B사 패키지 상세 → 같은 작업 반복
3. C사 패키지 상세 → 같은 작업 반복
4. D사 패키지 상세 → 같은 작업 반복
5. 자사 패키지 상세 → 같은 작업 반복
```

5번 같은 작업. 마커가 200개이고 패키지가 10개면 최악의 경우 2,000번이다.

그리고 더 위험한 건, **하나를 빠뜨려도 모른다는 것**이다. A사는 OR 1.42로 바뀌었는데 C사는 여전히 1.35라면, 같은 유전자 변이에 대해 벤더마다 다른 점수가 나온다. 이건 데이터 불일치를 넘어서 검사 신뢰도 문제다.

## 3. 정규화 본능: 마스터 테이블을 분리하자

데이터베이스를 배운 사람이라면 이 상태를 보고 본능적으로 떠올리는 게 있다. **정규화**.

같은 데이터가 여러 행에 복제되어 있다 → 마스터 테이블로 빼고 FK로 참조하자.

설계를 시작했다.

### 3계층 구조

```
[Layer 1] snp_marker — SNP 과학적 사실 (rs_number + gene 복합키)
┌────┬────────┬──────┬──────┬──────┬────────┐
│ id │ rs_num │ gene │ risk │ OR   │ Beta   │
│────│────────│──────│──────│──────│────────│
│ 1  │ rs6152 │ AR   │ G    │ 1.35 │ 0.3001 │  ← 전 세계에서 1행
└────┴────────┴──────┴──────┴──────┴────────┘

[Layer 2] snp_probeset_map — 칩 ↔ SNP 매핑
┌────┬──────────────┬────────────┬────────┐
│ id │ probeset_id  │ snp_marker_id│ strand │
│────│──────────────│────────────│────────│
│ 1  │ AX-11161415  │ 1 (rs6152) │ F      │
└────┴──────────────┴────────────┴────────┘

[Layer 3] plan_snp_weight — 패키지별 설정
┌────┬─────────┬────────────┬──────┬──────────┬──────────┐
│ id │ plan_id │ snp_marker_id│ type │ category │ OR오버라이드│
│────│─────────│────────────│──────│──────────│──────────│
│ 1  │ 1 (A사) │ 1 (rs6152) │ 탈모1│ TYPE1-T  │ NULL     │  ← 기본값 사용
│ 2  │ 3 (C사) │ 1 (rs6152) │ 탈모 │ 탈모전체 │ "1.30"   │  ← C사만 자체값
└────┴─────────┴────────────┴──────┴──────────┴──────────┘
```

오버라이드 패턴도 설계했다. 대부분의 벤더는 기본 OR/Beta를 쓰되, 자체 연구값이 있는 벤더만 덮어쓰는 구조.

```java
effectiveOR = plan_snp_weight.or_val_override ?? snp_marker.or_val
```

깔끔했다. `rs_number + gene`이 복합 유니크 키이고, 칩 매핑은 별도 테이블이고, 패키지별 설정은 오버라이드 패턴으로. 교과서적인 정규화.

기분이 좋았다. 잠깐.

## 4. 멈추고 생각했다

설계를 다 해놓고 나서 한 가지가 걸렸다.

> 이 정규화가 실제로 해결하는 문제가 뭐지?

정리하면:

- **해결하는 문제**: rs6152의 OR을 바꿀 때 1행만 수정하면 된다.
- **발생하는 비용**: 테이블 2~3개 추가, 마이그레이션 스크립트, 계산 로직 전면 수정, JOIN 쿼리 변경, 테스트 전체 재작성.

그런데 잠깐.

지금 weighted_info에서 OR을 수정하는 쿼리는 이렇다:

```sql
UPDATE weighted_info SET or_val = '1.42', b_val = '0.3507'
WHERE id = 12345;
```

1건씩. `WHERE id = ?`. 이걸 5번 반복하는 게 문제였다.

그런데 이걸 이렇게 바꾸면?

```sql
UPDATE weighted_info SET or_val = '1.42', b_val = '0.3507'
WHERE rs_number = 'rs6152' AND gene = 'AR';
```

끝이다. **WHERE 절만 바꾸면 5행이 한 번에 수정된다.**

테이블 분리 없이. 마이그레이션 없이. 계산 로직 변경 없이.

## 5. 조삼모사를 알아차린 순간

테이블을 분리하는 이유를 다시 생각해봤다.

```
정규화의 목적:
"같은 데이터를 여러 곳에 저장하지 않으면 수정 시 일관성이 보장된다."

실제 우리 상황:
"같은 데이터가 여러 행에 있지만, 한 번의 UPDATE로 전부 바꿀 수 있다."
```

정규화는 **저장 시점에서 중복을 제거**하는 방법이다.
하지만 우리 문제는 저장이 아니라 **수정**이었다.

수정 문제를 수정 방식으로 풀면 되는 걸, 저장 구조를 바꿔서 풀려고 했던 거다.

이게 바로 조삼모사다.

```
방법 1 (정규화):
  저장 구조를 바꿔서 중복을 제거 → 수정이 자연스럽게 1회가 됨
  비용: 테이블 2~3개 추가, 마이그레이션, JOIN 변경, 테스트 재작성

방법 2 (수정 로직 변경):
  저장 구조는 그대로 → 수정 쿼리의 WHERE 절만 변경
  비용: API 1~2개 추가
```

결과는 같다. rs6152의 OR을 바꾸면 5개 패키지 모두 반영된다.
하지만 비용은 하늘과 땅 차이다.

## 6. 실제로 추가한 것

### 백엔드: API 2개

```java
// 1. rs + gene 기준으로 해당 마커가 포함된 패키지 목록 조회
GET /admin/gene/weighted-info/search-by-rs?rsNumber=rs6152&gene=AR

// 2. rs + gene 기준 일괄 수정
PUT /admin/gene/weighted-info/bulk-by-rs
{
    "rsNumber": "rs6152",
    "gene": "AR",
    "fields": {
        "orVal": "1.42",
        "bVal": "0.3507"
    }
}
```

### Repository: 메서드 2개

```java
// WeightedInfoRepository
List<WeightedInfo> findByRsNumberAndGene(String rsNumber, String gene);

@Modifying
@Query("UPDATE WeightedInfo w SET w.orVal = :orVal, w.bVal = :bVal " +
       "WHERE w.rsNumber = :rsNumber AND w.gene = :gene")
int bulkUpdateByRsAndGene(@Param("rsNumber") String rsNumber,
                          @Param("gene") String gene,
                          @Param("orVal") String orVal,
                          @Param("bVal") String bVal);
```

이게 전부다.

기존 테이블 변경 없음. 계산 로직 변경 없음. 마이그레이션 없음.

### 운영 흐름 변경

```
Before:
  논문 갱신 → 패키지 5개에 각각 들어가서 마커 찾아서 수정 (5회)

After:
  논문 갱신 → 마커 관리에서 rs6152 검색 → 일괄 수정 (1회)
  → 영향받는 패키지 5개 자동 표시 → 확인 → 적용
```

## 7. 그렇다면 정규화는 언제 해야 하나

정규화를 안 한 게 아니라, **지금은 안 한 것**이다.

현재 방식이 버티지 못하는 시점이 온다면 그때 분리하면 된다. 구체적으로:

| 상황 | 현재 방식으로 충분 | 정규화 필요 |
|------|------------------|-----------|
| OR/Beta 일괄 수정 | `WHERE rs_number AND gene` ✓ | - |
| 벤더별 OR 오버라이드 | 해당 plan_id만 별도 수정 ✓ | - |
| 칩 교체 (AX 번호 변경) | - | probeset_id ↔ rs 매핑 분리 필요 |
| 같은 rs인데 벤더마다 risk가 다름 | - | 구조적 분리 필요 |
| gene/rs_number 정합성 강제 | 운영 규칙으로 관리 | UK 제약으로 DB가 보장 |

**칩 교체**가 현실화되는 시점이 정규화의 트리거다. 같은 rs6152를 다른 칩에서 다른 probeset_id로 읽어야 하는 순간, `snp_probeset_map` 테이블이 필요해진다.

하지만 현재 칩은 1종류이고, 교체 계획도 없다. 존재하지 않는 문제를 미리 풀면 그건 엔지니어링이 아니라 과설계다.

---

## 8. 정규화 설계를 한 시간이 낭비였나

아니다. 오히려 반대다.

정규화 설계를 해봤기 때문에 **어디까지가 진짜 필요하고, 어디부터가 과설계인지** 경계를 알게 됐다.

설계를 안 하고 "그냥 WHERE 절 바꾸면 되지 않나?"라고 처음부터 결론 냈다면, 그건 판단이 아니라 감이다. 정규화했을 때의 구조, 테이블 수, 마이그레이션 비용, 계산 로직 변경 범위를 다 그려봤기 때문에 "지금은 아니다"라는 결론에 근거가 생겼다.

설계 문서도 버리지 않았다. 3계층 구조, 오버라이드 패턴, DDL, 마이그레이션 전략까지 남겨뒀다. 나중에 칩 교체가 현실화되면 이 문서를 꺼내서 쓰면 된다.

> 설계는 구현하지 않더라도 가치가 있다.
> "이 설계를 왜 구현하지 않았는가"까지 남기면 더 가치가 있다.

## 9. 돌아보며

정규화를 가르칠 때 보통 이렇게 말한다.

"같은 데이터가 여러 곳에 있으면 정규화하라."

맞는 말이다. 하지만 이 원칙을 기계적으로 적용하면, **문제의 본질을 놓칠 수 있다.**

이번에 내가 겪은 것이 정확히 그랬다.

```
보이는 것: 데이터 중복 (같은 rs가 5행에 복제)
떠오른 해법: 정규화 (마스터 테이블 분리)
실제 문제: 수정 방식이 건별로만 가능했던 것
실제 해법: 수정 쿼리의 WHERE 절 변경
```

데이터 중복이 문제인 게 아니었다. 중복된 데이터를 **한 번에 수정할 방법이 없었던 것**이 문제였다.

저장 구조의 문제인지, 수정 방식의 문제인지. 이 구분 하나가 테이블 3개 추가와 API 2개 추가의 차이를 만들었다.

## 10. 정리

유전자검사 서비스의 가중치 테이블을 보면서 정규화 충동이 왔다.

3계층 테이블 구조를 설계하고, 오버라이드 패턴을 만들고, DDL을 작성하고, 마이그레이션 전략까지 세웠다. 깔끔했다.

그런데 한 발짝 물러서 보니, 실제로 필요한 건 이거였다:

```sql
-- Before: 1건씩 수정
UPDATE weighted_info SET or_val = '1.42' WHERE id = ?;
-- × 5회 반복

-- After: rs+gene 기준 일괄 수정
UPDATE weighted_info SET or_val = '1.42' WHERE rs_number = 'rs6152' AND gene = 'AR';
-- 1회로 끝
```

테이블 분리는 하지 않았다. 대신 설계 문서는 남겨뒀다. 칩이 바뀌거나 벤더별 OR 오버라이드가 빈번해지는 날이 오면, 그때 꺼내 쓸 생각이다.

> 좋은 설계는 구현된 설계가 아니라 적절한 시점에 구현되는 설계다.
> 그리고 그 시점을 아는 것이 설계보다 어렵다.
