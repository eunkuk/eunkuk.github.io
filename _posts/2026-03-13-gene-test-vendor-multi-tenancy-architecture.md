> 유전자검사 서비스에 벤더사가 붙기 시작했다. "멀티테넌시를 해야 하나?"가 떠올랐고, 개념을 정리한 뒤 실제 우리 상황에 맞는 수준을 골랐다. 풀 멀티테넌시가 아니라 점진적 B2B 오버레이를 선택한 이유를 기록한다.

---

## 1. 상황

> 이 글은 [PHP CodeIgniter 프로젝트를 Spring Boot로 옮기기까지 — 마이그레이션 회고 (1) 왜 다시 만들었나](/posts/php-codeigniter-spring-boot-migration-why/)와 [유전자검사 결과 저장: JSON → EAV 마이그레이션 회고](/posts/eav-vs-jsonb-gene-data-retrospective/)의 연장이다. 플랜 개념을 도입하면서 단일 상품 구조를 걷어냈고, 검사 결과 저장도 유연하게 바꿨다. 그런데 이제는 다른 종류의 요구가 붙기 시작했다. **벤더사별로 같은 유전자검사 플랫폼을 써야 하는 상황**이다.

지금까지는 전제가 단순했다.

- 서비스 운영사 1개
- 관리자 조직 1개
- 회원 풀 1개
- 결과지 체계 1개
- 플랜/정산/브랜딩 규칙 1개

이 전제에서는 하드코딩이 생각보다 오래 버틴다. 로고도 하나면 되고, 관리자 권한도 단순하다. 결과지도 한 벌이면 된다.

그런데 벤더사가 붙기 시작하면 상황이 달라진다.

- 벤더 A는 자기 로고와 브랜드명으로 결과지를 내고 싶다
- 벤더 B는 같은 검사 엔진을 쓰지만 검사 항목 구성이 다르다
- 벤더 C는 BASIC/PRO가 아니라 자기 상품 체계를 따로 가져가고 싶다
- 각 벤더의 이용권과 검사 내역은 섞이면 안 된다

여기서 핵심은 "기능이 조금 추가됐다"가 아니라는 점이다.
**운영 주체가 하나에서 여러 개로 바뀌기 시작하는 것**이다.

이 시점에서 떠오른 키워드가 멀티테넌시였다.

---

## 2. 멀티테넌시란 무엇인가

멀티테넌시는 **하나의 애플리케이션이 여러 고객 조직(tenant)을 동시에 수용하는 구조**다.

여기서 tenant는 보통 회사, 기관, 브랜드, 파트너사 같은 "독립 운영 단위"를 뜻한다.

| 구분 | Single Tenant | Multi Tenant |
|------|---------------|--------------|
| 운영 대상 | 고객사 1개당 시스템 1벌 | 시스템 1벌로 고객사 여러 개 |
| 코드베이스 | 고객사별로 따로 갈라지기 쉬움 | 공통 코드베이스 유지 |
| 데이터 | 물리적으로 쉽게 분리 | 논리적 분리 설계가 핵심 |
| 운영 비용 | 고객 늘수록 선형 증가 | 공통 운영 가능 |
| 위험 | 배포/운영 중복 | 격리 실패가 곧 보안 사고 |

핵심은 단순하다.

```
코드는 공유한다.
데이터와 설정과 권한은 테넌트별로 분리한다.
```

멀티테넌시는 "SaaS 회사들이 쓰는 거대한 패턴"처럼 보이지만, 본질은 거창하지 않다.
**같은 코어를 여러 조직이 공유하기 시작하면 결국 필요한 구조**다.

---

## 3. 벤더 = 테넌트인가

이건 먼저 구분해야 한다.
벤더사가 붙었다고 해서 무조건 멀티테넌시가 필요한 건 아니다.

예를 들어 이런 경우는 테넌트가 아닐 수 있다.

- 벤더가 시약이나 장비만 공급한다
- PDF 템플릿만 몇 장 납품한다
- 내부 운영 조직은 여전히 하나다
- 회원, 주문, 관리자 권한, 정산이 전부 한 회사 기준이다

이 경우 벤더는 그냥 **외부 메타데이터 공급자**에 가깝다.
`vendor_code` 정도로 구분하고 끝낼 수도 있다.

반대로 아래 조건이 붙으면 사실상 tenant다.

- 벤더별 관리자 조직이 따로 있다
- 벤더별 회원/주문/검사 내역이 섞이면 안 된다
- 벤더별 상품 구성과 가격 정책이 다르다
- 벤더별 결과지, 브랜딩, 약관, 안내 문구가 다르다
- 벤더별 정산과 통계를 따로 봐야 한다

이건 "벤더 지원"이 아니라 **하나의 플랫폼 위에 여러 운영 주체가 올라오는 구조**다.

> 이 벤더가 독립된 운영 단위인가?
> 데이터와 권한과 설정을 분리해야 하는가?

여기에 YES가 나오면 멀티테넌시다.

### 우리 상황의 진단

그래서 우리 벤더는 지금 어디에 있는가?

솔직히 답하면 **아직 완전한 테넌트는 아니다.**

- 벤더별 관리자 조직이 따로 있나? → 아직 없다. 우리 Admin이 대행한다.
- 벤더별 회원이 있나? → 없다. B2B 대상자는 이용권에 스냅샷으로 붙는다.
- 벤더별 상품 구성이 다른가? → 다르다. 플랜과 검사 항목이 벤더마다 다르다.
- 벤더별 결과지가 다른가? → 다르다. 템플릿이 벤더마다 다르다.
- 벤더별 정산이 있나? → 아직 아니다.

5개 중 2개만 YES다. 나머지는 아직 우리 조직이 대행하고 있다.
이 상태에서 풀 멀티테넌시를 도입하면 **현재 없는 문제를 미리 해결하는 과설계**가 된다.

## 4. 왜 유전자검사 서비스에서 특히 주의해야 하나

그렇다고 멀티테넌시를 완전히 무시할 수도 없다.
유전자검사 서비스는 겉보기보다 테넌트별 차이가 많이 붙는 구조다.

### 브랜딩만 다른 게 아니다

처음엔 로고, 컬러, 결과지 표지 정도만 다를 거라고 생각하기 쉽다.
그런데 실제로는 그 뒤에 붙는 설정이 훨씬 많다.

- 검사 상품 이름
- 플랜별 검사 항목 구성 (SNP 세트, 가중치, 등급 기준)
- 결과지 문구와 참고 문헌
- 관리자 권한 범위
- 정산 기준과 대시보드 지표

결과적으로 "같은 검사 엔진" 위에 **다른 서비스 껍데기**가 여러 개 올라간다.

### 데이터 격리가 곧 신뢰다

벤더 A의 관리자가 벤더 B의 검사 내역을 볼 수 있다면 그건 기능 버그가 아니라 사고다.
유전자검사 데이터는 민감하다. 대상자 정보, 검체 번호, 검사 결과, PDF 결과지까지 벤더 간에 섞이면 안 된다.

### 코드 포크로 가면 금방 무너진다

벤더가 2곳일 때는 "A사 전용 분기" 정도로 버틸 수 있다.
그런데 3곳, 4곳이 되면 이런 코드가 퍼진다.

```java
if (vendorCode.equals("A")) {
    // A사 전용 결과지
} else if (vendorCode.equals("B")) {
    // B사 전용 상품 노출
} else if (vendorCode.equals("C")) {
    // C사 전용 문구
}
```

이 방식은 벤더 수가 아니라 **변형 포인트 수**에 비례해서 무너진다.
이용권, PDF, 정산, 어드민, 통계마다 분기가 생기면 사실상 고객사별 포크와 다를 게 없어진다.

## 5. 풀 멀티테넌시의 이상적인 모습

참고로 풀 멀티테넌시를 적용한다면 어떤 구조가 되는지 정리해둔다.
지금 바로 하겠다는 게 아니라, **나중에 어디까지 갈 수 있는지 그림을 알아야** 현재 설계에서 막다른 길을 피할 수 있기 때문이다.

### 테넌트 소유 데이터 vs 공통 마스터

```
공통 마스터: 공유          → SNP 메타, 유전자 해석 엔진, 공통 Enum
운영 데이터: tenant별 분리  → 회원, 이용권, 검사결과, 플랜, 결과지
표현 메타: tenant별 override → 브랜딩, 문구, 약관
```

풀 멀티테넌시라면 아래 테이블에 전부 `tenant_id`가 들어간다.

```sql
member, voucher, gene, gene_result, plan, report_template, admin_user ...
```

### 구현 방식 세 가지

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| DB per Tenant | 벤더마다 DB를 따로 둔다 | 격리 강함 | 운영 비용 큼, 마이그레이션 번거로움 |
| Schema per Tenant | DB 하나, 스키마를 테넌트별 분리 | DB 격리와 운영 효율의 중간 | 스키마 관리 복잡 |
| Shared DB + `tenant_id` | 같은 테이블을 쓰고 행 단위로 분리 | 가장 단순하고 확장 빠름 | 쿼리 누락 시 격리 사고 |

### 애플리케이션 계층

```
Request
  ↓
TenantResolver (subdomain / header / token)
  ↓
TenantContext (ThreadLocal or Scoped)
  ↓
Facade / DomainService
  ↓
Repository (WHERE tenant_id = ?)
```

이 구조까지 가면 벤더별 격리가 완전하다. 하지만 기존 B2C 테이블을 전부 건드려야 하고, 인증 체계도 바꿔야 한다.

**지금 이걸 다 할 필요가 있는가?** 가 핵심 질문이었다.

## 6. 우리가 선택한 접근: B2B 오버레이

결론부터 말하면 **풀 멀티테넌시는 채택하지 않았다.**
대신 기존 B2C를 유지한 채, B2B 전용 테이블을 별도로 올리는 방식을 선택했다.

### 기존 B2C는 건드리지 않는다

```
B2C (기존, 변경 없음)              B2B (신규)
─────────────────────            ─────────────────
member                           vendor
voucher                          b2b_voucher
voucher_kit                      b2b_voucher_kit
```

`member`, `voucher`, `voucher_kit`에 `tenant_id`를 추가하는 대신, B2B는 아예 다른 테이블로 분리했다. 이유는 단순하다.

- 기존 B2C 코드에 영향 제로
- B2B 전용 생명주기(발급→키트→검사→완료)가 B2C와 다름
- 기존 공통 조회/리포트 경로가 `VoucherKit → Voucher → Member` 전제를 갖고 있어서, `tenant_id`를 붙여도 B2B 경로는 따로 만들어야 함

### Plan이 핵심 축이다

벤더마다 검사 항목, 등급 기준, SNP 가중치가 다른 건 결국 **Plan 하나 만들면 해결되는 구조**다.

기존 `plan_gene_config`, `plan_gene_grade_config`, `weighted_info`는 이미 `plan_id` FK로 스코프가 나뉘어 있다. 벤더사별로 Plan을 생성하고, 그 Plan에 설정을 붙이면 끝이다.

그래서 기존 `plan` 테이블에 `vendor_id` 컬럼 하나만 추가했다. 이것이 유일한 기존 테이블 변경이다.

```sql
ALTER TABLE plan ADD COLUMN vendor_id BIGINT NOT NULL;
```

`plan_type`(BASE_PLAN/ADDON)은 플랜의 구조적 역할을 구분하는 축이고, `vendor_id`는 플랜의 소유자를 구분하는 축이다. 이 두 축은 독립적이다.

### 우리 자신도 벤더로 등록한다

한 가지 더 결정한 것이 있다. 회사 자체를 `vendor` 테이블에 등록했다.

```
vendor
├── id=1, code="회사", type=COMPANY  ← 우리
├── id=2, code="DIST001",   type=DISTRIBUTOR ← 총판
└── id=3, code="HANI001",   type=CLINIC      ← 병원
```

이렇게 하면 `plan.vendor_id`가 **항상 NOT NULL**이 된다. "내 서비스인지 외부 벤더인지"를 nullable 분기로 판단하는 게 아니라, 모든 플랜이 동일한 구조를 갖게 된다. "벤더별 플랜 목록", "벤더별 이용권 집계" 같은 쿼리도 내 서비스를 포함해서 같은 패턴으로 동작한다.

### 벤더 계층은 parent_id로 표현한다

총판(1차) → 병원(2차) 판매 구조를 위해 `vendor` 테이블에 `parent_id` self-referencing을 넣었다. 기존 코드베이스의 `Menu` 엔티티가 같은 패턴을 쓰고 있어서 검증된 방식이다.

계층이 1단계(총판→병원)로 고정이라 재귀 쿼리 없이 단순 join이면 충분하다.

## 7. 이 접근이 멀티테넌시와 갈라지는 지점

풀 멀티테넌시와 비교하면 우리 접근은 이렇게 다르다.

| 항목 | 풀 멀티테넌시 | B2B 오버레이 (우리 선택) |
|------|-------------|----------------------|
| 기존 테이블 변경 | member, voucher 등에 `tenant_id` 추가 | **변경 없음** (plan에 vendor_id만 추가) |
| B2B 데이터 | 기존 테이블 행 단위 분리 | **별도 테이블** (b2b_voucher, b2b_voucher_kit) |
| 인증 | TenantResolver, JWT tenantId | Admin이 대행 (벤더 로그인 없음) |
| 리포트 | 공통 resolver + tenant 분기 | **B2B 전용 resolver 신규 작성** |
| 회원 | 기존 member에 tenant_id | B2B에 회원 개념 없음 (스냅샷) |
| 확장 비용 | 높음 (전면 리팩토링) | 낮음 (신규 테이블 추가) |
| 격리 수준 | 완전 (모든 쿼리에 tenant 조건) | 구조적 (테이블 자체가 분리) |

핵심 트레이드오프는 이거다.

- 풀 멀티테넌시는 **하나의 통합된 데이터 모델** 위에 tenant 축을 넣는다. 깔끔하지만, 기존 코드 전체에 영향이 간다.
- B2B 오버레이는 **기존 모델은 유지하고 옆에 B2B 모델을 세운다**. 기존 코드 영향 제로지만, 두 벌의 모델이 생긴다.

지금 단계에서는 후자가 맞다고 판단했다. 기존 B2C가 안정적으로 돌아가고 있고, B2B 벤더 수가 아직 적고, 관리도 Admin이 대행하는 상황이기 때문이다.

## 8. 그래도 멀티테넌시를 의식하고 설계한 부분

풀 멀티테넌시를 채택하지 않았다고 해서, 나중에 가는 길을 막아놓은 건 아니다.
오히려 의식적으로 **확장 가능한 구조**를 만들어놨다.

### vendor_id를 Plan에 넣은 것

나중에 `member`, `voucher`에도 `vendor_id`가 필요해지면, `plan.vendor_id` 패턴을 그대로 확장하면 된다. nullable `tenant_id`가 아니라 NOT NULL `vendor_id`로 시작했기 때문에 패턴이 깨끗하다.

### 자체를 벤더로 등록한 것

"우리 = vendor id 1"이라는 전제가 있으면, 나중에 모든 테이블에 `vendor_id`를 추가할 때 기존 데이터 마이그레이션이 `UPDATE SET vendor_id = 1` 한 줄로 끝난다.

### 결과지 경로를 분리한 것

B2B 리포트를 기존 공통 resolver에 억지로 끼워 넣지 않고, 별도 resolver를 만들었다. 기존 `VoucherKit → Voucher → Member` 전제를 깨지 않으면서 B2B 전용 경로 `B2bVoucherKit → B2bVoucher → Vendor → Plan`을 세웠다.

나중에 벤더별 결과지 전략이 복잡해지면, 이 분리된 경로 위에 전략 패턴을 올리면 된다.

```java
// 지금은 vendor.report_template_code로 템플릿만 분기
// 나중에 필요하면 전략 인터페이스로 확장 가능
public interface B2bReportPolicy {
    boolean supports(String reportTemplateCode);
    ReportRenderData assemble(B2bReportSource source);
}
```

### vendor_code와 template_code를 분리한 것

`vendor_code`는 조직 식별자이고, `report_template_code`는 리포트 템플릿 식별자다. 이 두 개를 합쳐놓으면 "벤더 A인데 벤더 B 템플릿을 쓰고 싶다"는 요구가 오는 순간 깨진다. 처음부터 분리했다.

## 9. 언제 풀 멀티테넌시로 전환해야 하나

지금 구조가 영원히 버틸 수는 없다. 아래 조건이 붙기 시작하면 전환 시점이다.

1. **벤더가 직접 로그인**해서 자기 데이터를 조회해야 할 때
2. **벤더별 회원 풀**이 생겨서 B2C member와 구조가 겹치기 시작할 때
3. **벤더별 정산과 통계**를 시스템 레벨로 분리해야 할 때
4. B2B 테이블과 B2C 테이블 사이에 **중복 로직**이 눈에 띄게 쌓일 때
5. 벤더 수가 늘어서 **별도 테이블 관리가 오히려 비효율**이 될 때

이때 전환 경로는 대략 이렇다.

```
1. vendor 테이블을 tenant 마스터로 승격
2. 기존 B2C 루트 테이블에 vendor_id 추가 (기존 데이터는 회사 id로 마이그레이션)
3. B2B 별도 테이블을 기존 테이블로 통합하거나, 둘 다 유지하며 조회 계층 통합
4. 인증에 tenant context 추가 (JWT, 서브도메인 등)
5. 결과지/브랜딩/상품 정책을 tenant 메타데이터 + 전략 객체로 분리
```

중요한 건, 지금 선택이 이 경로를 막지 않는다는 것이다.
회사가 이미 vendor로 등록되어 있고, Plan에 vendor_id가 들어가 있으니, 2번 단계의 마이그레이션 비용이 크지 않다.

## 10. 정리

유전자검사 서비스에 벤더사가 붙는다는 건, 단순히 파트너가 하나 늘었다는 뜻이 아니다.
**운영 모델이 single-tenant에서 multi-tenant로 바뀌기 시작할 수 있다는 신호**다.

하지만 신호를 봤다고 바로 풀 멀티테넌시를 구현할 필요는 없다.

우리가 내린 판단은 이렇다.

1. **벤더는 아직 완전한 테넌트가 아니다.** 관리자 조직, 회원, 정산이 아직 우리 기준이다.
2. **기존 B2C를 건드리지 않는 B2B 오버레이가 현실적이다.** 별도 테이블 3개 + Plan에 vendor_id 하나.
3. **나중에 풀 멀티테넌시로 가는 길은 열어뒀다.** 회사 자체 벤더 등록, vendor_id NOT NULL 패턴, 리포트 경로 분리.

> 멀티테넌시는 "할 것인가 말 것인가"의 이분법이 아니다.
> "지금 어디까지 필요한가"를 정확히 재고, 나중에 확장할 수 있는 구조를 남겨두는 것이다.

풀 멀티테넌시를 하지 않았다고 해서 설계를 대충 한 게 아니다.
오히려 **과설계를 피하면서 확장 경로를 확보하는 것**이 더 어려운 판단이었다.
