# 같은 도메인, 다른 결과지 — Transfer Layer로 표현 계층 분리하기

첫 번째 PDF 결과지를 만들 때는 보통 도메인 데이터만 잘 꺼내오면 끝난다.  
문제는 두 번째 결과지가 붙는 순간 시작된다.

"이미 유전자 검사 결과 데이터는 있는데, 결과지 구조가 완전히 다르다."

이 상황에서 선택지는 두 가지다.  
Entity를 수정해서 새 구조를 집어넣거나, 중간 계층을 둬서 같은 Entity를 다른 모델로 변환하거나.  
나는 후자를 택했고, 그 이유와 결과를 정리해봤다.

---

## 1. Entity는 공통인데 PDF는 공통이 아니었다

처음엔 같은 유전자 검사 결과니까 같은 모델로 밀어도 되지 않을까 생각했다.  
그런데 조금만 뜯어보면 전혀 다르다.

공통으로 필요한 데이터는 분명 있다.

- 사용자 이름 / 생년 / 성별
- 검체 번호
- 검사 결과 등급
- SNP 메타 정보 및 유전자형

하지만 PDF가 요구하는 **구조**는 결과지마다 달랐다.

**A 타입 결과지**는 항목 중심으로 페이지를 전개한다.
```
종합 요약 → 부모 항목 → 자식 항목 → SNP 상세
```

**B 타입 결과지**는 장기/카테고리 중심이다.
```
장기별 요약 → 장기 안의 카테고리 → 카테고리 소개 → 상세 결과
```

여기서 중요한 건, 이 차이가 **엔티티 차이가 아니라 표현 구조 차이**라는 점이다.  
도메인 데이터는 크게 달라지지 않았다.  
달라진 건 "그 데이터를 어떤 기준으로 묶고, 어떤 순서로 보여주느냐"였다.

---

## 2. Entity를 건드리지 않은 이유 — "의미 오염"

Entity를 건드리는 순간 공수가 커진다.  
하지만 그보다 더 큰 문제는 **의미 오염**이다.

예를 들어 B 타입 결과지가 장기 기준 분류를 쓴다고 해서 `PlanGeneConfig`에 장기 필드를 넣기 시작하면, 그 순간부터 도메인 모델은 특정 보고서 형식의 요구를 떠안게 된다.  
결과지 C가 다른 분류 기준을 쓰면 필드가 하나 더 붙고, D가 또 다른 기준을 쓰면 또 붙는다.

처음엔 "유연하게 대응했다"고 느껴지지만, 몇 번만 반복되면 **Entity가 보고서 렌더링 설정 저장소**처럼 변한다.

내가 피하고 싶었던 건 정확히 그 상태였다.

```
Entity는 공통 원천 데이터에만 집중한다.
결과지 구조는 결과지별 표현 모델로 분리한다.
둘 사이를 이어주는 계층을 둔다.
```

그 계층을 나는 **Transfer Layer**라고 불렀다.

---

## 3. Transfer Layer의 구조

이 계층이 하는 일은 단순하다.  
공통 도메인 데이터를 한 번 모은 다음, 결과지별 PDF 모델로 변환하는 것이다.

```
Entity / Domain Service
    ↓
CommonReportSource       ← 도메인 친화적
    ↓
VendorReportAssembler    ← 변환 지점
    ↓
VendorReportData         ← 렌더링 친화적
    ↓
HTML Builder / PDF Renderer
```

코드로 옮기면 대략 이런 형태다.

```java
public record CommonReportSource(
        Member member,
        Gene gene,
        Plan plan,
        Voucher voucher,
        Map<String, GeneResultValueGrade> grades,
        List<PlanGeneConfig> configs,
        GeneTestCalculationResult calcResult
) {}
```

```java
public interface VendorReportAssembler<T extends VendorReportData> {
    boolean supports(String vendorCode);
    T assemble(CommonReportSource source);
}
```

결과지 타입마다 Assembler 구현체를 하나씩 둔다.  
`CommonReportSource`는 도메인 친화적이고, `VendorReportData`는 렌더링 친화적이다.  
둘의 책임이 섞이지 않도록 끊어주는 게 이 계층의 핵심이다.

---

## 4. 메타데이터는 어떻게 관리할 것인가

Transfer Layer를 뒀다고 해서 모든 걸 동적으로 만든 건 아니다.  
오히려 초반엔 반대로 갔다.

공통 원천 데이터는 재사용하되, **결과지별 표현 메타데이터는 상당 부분 하드코딩**했다.

- 장기 색상 / 아이콘
- 카테고리명 / 설명문
- 결과 문구 / 참고문헌
- 이미지 경로 / 페이지 순서

이건 도메인 데이터가 아니라 **보고서 스킨**에 가깝기 때문이다.  
이런 영역까지 처음부터 CMS처럼 만들면 구현 속도는 느려지고 구조는 오히려 무거워진다.

메타데이터 관리 방식은 크게 세 가지가 있고, 각각 트레이드오프가 다르다.

| | 하드코딩 | DB 테이블 | 외부 파일 (JSON/YAML) |
|---|---|---|---|
| **패턴 명칭** | Static Metadata / Constant Registry | Metadata Table Pattern | File-based CMS / Config-driven |
| **변경 방법** | 코드 수정 + 재배포 | SQL UPDATE | 파일 교체 |
| **타입 안정성** | ✅ 컴파일 타임 체크 | ❌ 런타임에서야 발견 | △ 별도 스키마 검증 필요 |
| **비개발자 수정** | ❌ | ✅ 어드민 UI 가능 | △ 파일 직접 편집 |
| **실수 가능성** | 낮음 | 높음 | 중간 |
| **적합한 데이터** | 아이콘, 색상, 페이지 순서 | 자주 바뀌는 문구 | 설명문, 가이드, 참고문헌 |

내가 실제로 선택한 건 **변경 빈도 기준으로 나누는 혼합 전략**이었다.

```
변경 빈도 낮음 → 하드코딩   (아이콘, 색상, 페이지 구조)
변경 빈도 중간 → 외부 JSON  (설명문, 가이드, 참고문헌)
변경 빈도 높음 → DB         (사용자별 score, 등급)
```

이 판단이 사실 Transfer Layer 설계만큼 중요한 의사결정이었다.

---

## 5. 하드코딩의 위치가 중요하다

하드코딩 자체가 나쁜 건 아니다.  
문제는 **하드코딩의 위치**다.

나쁜 방식은 이렇다.

```java
if (vendorCode.equals("TYPE_B")) {
        // 설명문, 색상, 아이콘, 카테고리, 페이지 순서 전부 여기에
        }
```

이렇게 되면 Builder가 도메인도 알고 메타데이터도 알고 렌더링도 한다.  
책임이 한 곳에 몰린다.

좋은 방식은 상수를 vendor별 메타데이터로 모아두는 것이다.

```
TypeBMetadataRegistry    ← 상수 모음
TypeBReportAssembler     ← 변환 로직
TypeBReportHtmlBuilder   ← 렌더링만
```

상수는 써도 된다.  
다만 그 상수가 Builder 안에 흩어지면 안 되고, **결과지별 메타데이터로 모여** 있어야 한다.

---

## 6. 결과적으로 얻은 것

**Entity 공수를 막았다**  
결과지 하나 추가될 때마다 Entity를 수정하지 않아도 됐다.  
공통 도메인은 그대로 두고, 중간 계층과 메타데이터만 건드리면 됐다.

**PDF Builder가 단순해졌다**  
HTML Builder는 이제 도메인을 이해할 필요가 없다.  
받은 DTO를 렌더링하는 데만 집중하면 된다.  
템플릿을 수정할 때도 JPA나 도메인 계산 로직을 같이 보지 않아도 된다.

**요구사항이 흔들려도 버텼다**  
이런 기능은 초반에 요구사항이 자주 바뀐다.

> "장기별 요약이 필요하다"  
> "카테고리 소개 페이지를 넣자"  
> "이 결과지는 문구를 다르게 쓰자"  
> "이미지 리소스를 바꾸자"

Entity를 건드리는 구조였다면 수정 범위가 너무 커졌을 거다.  
Transfer Layer 중심으로 끊어두니 수정 포인트가 제한됐다.

---

## 7. 실제로 결과지가 하나 더 붙었을 때

구조가 잘 잡혀있는지 확인하는 가장 좋은 방법은 실제로 결과지가 추가됐을 때 무슨 일이 일어나는지 보는 것이다.

C 타입 결과지가 새로 붙었다고 가정하자.

**새로 만든 것**
```
TypeCMetadataRegistry    ← 새 파일
TypeCReportAssembler     ← 새 파일
TypeCReportHtmlBuilder   ← 새 파일
typec.json               ← 새 파일 (메타데이터가 외부 파일인 경우)
```

**건드리지 않은 것**
```
Entity                   ← 그대로
CommonReportSource       ← 그대로
VendorReportAssembler    ← 인터페이스 그대로
TypeAReportAssembler     ← 그대로
TypeBReportAssembler     ← 그대로
```

새 결과지를 추가했는데 기존 코드를 한 줄도 수정하지 않았다.  
이게 이 구조가 제대로 잡혔다는 신호였다.

반대로 구조가 잘못 잡혔을 때는 이런 일이 생긴다.

```
새 결과지 추가
  → Entity에 필드 추가
  → 기존 Assembler에 분기 추가
  → Builder에 if 블록 추가
  → 기존 결과지 동작 확인 필요
```

추가인데 수정이 따라온다.  
그리고 기존 결과지가 영향을 받았는지 확인해야 하는 순간, 테스트 범위가 갑자기 넓어진다.

Transfer Layer를 제대로 끊어두면 **결과지 추가는 파일 추가**로 끝난다.  
이게 이 구조에서 실질적으로 가장 큰 이점이었다.

---

## 마무리

이번 작업에서 얻은 가장 큰 교훈은 이거였다.

> **공통화는 Entity를 키우는 방식이 아니라, 변환 지점을 명확하게 두는 방식으로 하는 게 더 오래 간다.**

같은 도메인 데이터를 여러 형태의 결과지로 보여줘야 한다면, 처음부터 모든 걸 범용 스키마로 만들려고 하기보다 이렇게 나누는 편이 낫다.

- 도메인 데이터는 공통으로 유지한다
- 결과지별 차이는 메타데이터와 조립 계층에서 처리한다
- PDF Builder는 렌더링에만 집중시킨다
- 메타데이터는 변경 빈도 기준으로 저장 방식을 나눈다

초기에는 파일이 조금 더 늘어나는 것처럼 보여도, 결과지가 하나 더 붙는 순간 그 차이가 바로 드러난다.
