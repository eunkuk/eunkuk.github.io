> 면접에서 `Strategy Pattern이 뭐냐`는 질문을 받았고, 순간 바로 대답하지 못했다. 그런데 설명을 듣는 순간 익숙한 코드가 떠올랐다. 추상화는 하고 있었지만 이름이 없었다. 그 경험을 계기로 패턴 암기, 설계 언어, LLM 시대의 활용에 대해 다시 생각하게 됐다.

---

## 빠른 요약

- 문제: 실무에서는 풀고 있던 구조를 면접에서는 패턴 언어로 바로 설명하지 못했다.
- 정리한 관점: 설계 감각과 설계 언어는 다르며, 패턴 이름은 설명, 검색, 검증, 협업을 위한 압축된 언어다.
- 결론: LLM이 회상 비용은 낮추지만, 지금 문제를 어떤 구조로 해석할지는 여전히 개발자의 몫이다.

---

## 1. 상황

면접에서 `Strategy Pattern이 뭐냐`는 질문을 받은 적이 있다.

순간 머리가 멈췄다.
"아, 그거 뭐였죠?"라는 반응이 먼저 나왔다.

면접관이 "같은 인터페이스 아래 구현을 바꿔 끼우는 방식"이라고 설명해주자 그제서야 익숙한 구조가 떠올랐다.

아, 이거.
실무에서 이미 비슷하게 하고 있었는데.

그때 떠올랐던 건 교과서 예제가 아니었다.
서비스 코드에서 분기문이 길어질 때 인터페이스를 만들고 구현체를 나누던 순간들이었다.

- 벤더별 결과지 조립 방식 분리
- 템플릿별 렌더링 정책 분리
- 조건별 처리 로직을 구현체로 나눠 꽂는 구조

예를 들어 [같은 도메인, 다른 결과지 — Transfer Layer로 표현 계층 분리하기](/posts/pdf-transfer-layer-commonization/)에서 정리했던 결과지 조립 구조나, [유전자검사 서비스에 벤더사가 붙기 시작했다 — 멀티테넌시를 고민할 시점](/posts/gene-test-vendor-multi-tenancy-architecture/)에서 적은 벤더별 리포트 정책 분리는 결국 "공통 인터페이스 + 구현체 선택" 구조였다.

문제는 그걸 **"이건 Strategy Pattern이다"라고 말하는 언어가 내 머릿속에 바로 안 붙어 있었다**는 점이다.

그 경험 이후에 든 생각은 단순했다.

> 추상화를 할 줄 아는 것과,
> 그 추상화를 패턴 언어로 설명할 수 있는 것은 다른 능력이다.

---

## 2. 이름 없이도 추상화는 할 수 있다

실무에서는 패턴 이름보다 문제가 먼저 온다.

문제는 보통 이렇게 시작한다.

- if-else가 길어진다
- 케이스가 계속 추가된다
- 공통 흐름은 같은데 세부 정책만 다르다
- 기존 코드를 매번 수정하는 게 불안해진다

이쯤 되면 자연스럽게 "이건 분기문으로 더 못 버티겠다"는 생각이 든다.
그리고 인터페이스를 만들고 구현체를 나누게 된다.

처음에는 대개 이런 식이다.

```java
if (templateCode.equals("A")) {
    return typeAAssembler.assemble(source);
} else if (templateCode.equals("B")) {
    return typeBAssembler.assemble(source);
} else if (templateCode.equals("C")) {
    return typeCAssembler.assemble(source);
}
```

그런데 템플릿이 늘어나고 분기 포인트가 여러 군데로 퍼지면 결국 이렇게 바뀐다.

```java
public interface ReportPolicy {
    boolean supports(String templateCode);
    ReportData assemble(CommonReportSource source);
}
```

```java
public class TypeAReportPolicy implements ReportPolicy {
    @Override
    public boolean supports(String templateCode) {
        return "A".equals(templateCode);
    }

    @Override
    public ReportData assemble(CommonReportSource source) {
        // A 타입 결과지 조립
    }
}
```

이걸 처음부터 "전략 패턴을 적용해야지" 하고 시작한 건 아니었다.
그냥 분기문이 버거워져서 그렇게 풀었을 뿐이다.

실무에서는 이런 일이 생각보다 많다.
패턴 이름은 몰라도, 문제를 밀어붙이다 보면 이미 그 패턴의 형태를 밟고 있는 경우가 많다.

---

## 3. 그런데 이름이 없으면 왜 불편한가

이름 없이도 구현은 가능하다.
하지만 이름이 없으면 그 지식은 개인 감각에 머무르기 쉽다.

### 설명이 약해진다

"조건문이 길어져서 클래스를 나눴다"와
"바뀌는 축을 Strategy로 분리했다"는 설명의 밀도가 다르다.

전자는 결과 설명이고,
후자는 **왜 그렇게 나눴는지에 대한 설계 의도**까지 담고 있다.

### 검색이 안 된다

이름이 없으면 비슷한 문제를 겪은 사람들의 글, 책, 사례를 찾기 어렵다.
검색어가 없기 때문이다.

"조건문 많을 때 클래스 나누기"로 찾는 것과
"strategy pattern when policy varies"로 찾는 것은 도달 가능한 정보량이 다르다.

### 검증이 안 된다

이름이 붙으면 비로소 비교가 가능해진다.

- 이건 정말 Strategy가 맞나
- 단순 enum 분기로 끝낼 수 있는 거 아닌가
- Factory가 더 맞는 문제 아닌가
- 추상화 시점이 너무 이른 거 아닌가

패턴 이름은 정답이 아니라, 설계를 검토하기 위한 좌표가 된다.

### 팀 언어가 생긴다

실무에서 중요한 건 혼자만 아는 감각이 아니다.
코드 리뷰, 설계 문서, 회의에서 같은 구조를 같은 말로 부를 수 있어야 한다.

패턴 이름이 필요한 이유는 멋있어 보이기 위해서가 아니라,
**설계를 공유 가능한 언어로 압축하기 위해서**다.

---

## 4. 그렇다고 패턴을 영단어처럼 외우는 게 답인가

여기서 반대로 이런 의문도 생긴다.

> 그럼 디자인 패턴을 영단어장처럼 외워야 하나?

내 생각은 "절반만 맞다"에 가깝다.

이름을 아는 건 분명 도움이 된다.
하지만 이름만 외우면 실무에서는 잘 안 보인다.

실무 문제는 보통 이렇게 오지 않는다.

- "지금부터 Strategy Pattern을 적용해야 하는 요구사항이 들어왔습니다"

실제 요구사항은 이렇게 온다.

- 벤더별로 결과지 생성 방식이 다르다
- 결제 수단별 검증 규칙이 다르다
- 채널별 알림 전송 정책이 다르다
- 상태별 후처리 로직이 다르다

즉, 문제는 패턴 이름으로 오지 않는다.
항상 **도메인 문제의 형태**로 온다.

그래서 이름만 외우는 건 약하다.
`Strategy`, `Factory`, `Adapter`, `Decorator`를 줄줄 말할 수 있어도,
언제 분기문으로 버텨야 하고 언제 추상화해야 하는지 모르면 실무에서는 별 힘이 없다.

중요한 건 패턴 이름 암기 자체가 아니라,
반복되는 문제를 보고 "이건 바뀌는 정책의 문제구나", "이건 생성 책임의 문제구나"라고 읽어내는 능력이다.

---

## 5. LLM 시대에는 더 안 외워도 되는가

요즘은 이 질문도 자연스럽다.

> 이제는 LLM한테 물어보면 되는데, 패턴 이름을 굳이 외워야 하나?

사실 맞는 말이다.
예전보다 회상 비용은 확실히 낮아졌다.

- "이 구조 이름이 뭐였지?"
- "인터페이스로 구현체 갈아끼우는 패턴 뭐였지?"
- "조건 분기를 클래스로 분리하는 대표 패턴이 뭐지?"

이런 건 이제 몇 초면 다시 찾을 수 있다.

하지만 LLM이 싸게 만든 건 **지식 접근 비용**이지,
**설계 판단 비용**은 아니다.

LLM은 이런 건 잘해준다.

- 지금 구조가 어떤 패턴과 닮았는지 설명
- 패턴 후보 여러 개 제시
- 각 패턴의 장단점 요약
- 예제 코드 생성

그런데 이런 건 대신 못 정한다.

- 지금 이 문제를 정말 추상화해야 하는가
- 아직 분기문으로 버티는 게 맞는가
- 변경 축이 충분히 안정적인가
- 구현체를 나누는 순간 복잡도가 더 커지지 않는가
- 지금 필요한 건 Strategy인가, 아니면 그냥 메서드 분리인가

즉, LLM이 있어도 결국 남는 질문은 이것이다.

> 지금 눈앞의 문제를 어떤 구조로 해석할 것인가?

이건 여전히 개발자의 몫이다.

---

## 6. 그 면접 이후에 바뀐 것

그 면접 이후로 패턴을 보는 방식이 조금 바뀌었다.

예전에는 "문제가 생기면 일단 추상화한다"에 가까웠다면,
이후에는 "이 추상화의 이름이 뭔지, 어떤 범주의 문제인지"를 한 번 더 생각하게 됐다.

예를 들어 지금은 이런 식으로 본다.

- 구현체 선택이 바뀌는가 → Strategy 가능성
- 객체 생성 방식이 문제인가 → Factory 가능성
- 공통 흐름 안의 일부 단계만 바뀌는가 → Template Method 가능성
- 기존 인터페이스를 맞춰 끼워야 하는가 → Adapter 가능성

이렇게 이름을 붙여서 생각하면 좋은 점이 있다.
설계가 갑자기 정교해진다기보다, **설계를 설명하고 비교하고 되짚는 속도**가 빨라진다.

그리고 LLM을 쓸 때도 질문이 더 좋아진다.

예전에는
"조건문이 많은데 클래스로 나누는 게 맞나?" 수준이었다면,

지금은
"이 문제는 Strategy와 Template Method 중 어느 쪽이 더 맞나?"
"현재 변경 축이 정책인지 절차인지 판단 기준은 뭔가?"
처럼 더 구조적인 질문을 던질 수 있다.

LLM이 강해질수록 오히려
**질문의 품질이 설계 품질을 좌우한다**는 걸 더 자주 느낀다.

---

## 7. 정리

면접에서 Strategy Pattern을 바로 설명하지 못했던 건,
내가 추상화를 몰라서가 아니었다.

**설계 감각은 있었지만, 그 감각을 설명하는 설계 언어가 부족했던 것**에 가까웠다.

그 경험 이후에 내린 결론은 단순하다.

- 패턴을 영단어처럼 외우기만 하는 것도 약하다
- 반대로 이름은 몰라도 된다고 치부하는 것도 약하다

패턴 이름은 정답이 아니라,
반복되는 설계 문제를 빠르게 공유하고 검토하기 위한 **압축된 언어**다.

그리고 LLM이 그 언어를 더 쉽게 꺼내주게 된 시대일수록,
오히려 중요한 건 이름을 말하는 능력보다
**지금 눈앞의 문제를 어떤 구조로 읽을 것인가를 판단하는 능력**이다.

> LLM은 패턴 이름을 싸게 알려준다.
> 하지만 어떤 문제에 그 이름을 붙여야 하는지는 아직 개발자의 일이다.
