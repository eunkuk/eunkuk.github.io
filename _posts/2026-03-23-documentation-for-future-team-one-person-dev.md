> 연차가 쌓이면서 이제는 코드만 잘 짜는 것보다 조직 운영과 지식 전파 쪽에도 관심이 생겼다. 그런데 현실은 비IT 회사의 1인 개발자다. 사람은 아직 없지만, 미래의 팀을 위해 지금부터 남겨야 할 문서가 무엇인지 정리해본다.

---

## 1. 상황

요즘 들어 개발을 보는 관점이 조금 바뀌고 있다.

예전에는 "어떻게 잘 만들 것인가"가 중심이었다면,
지금은 "이 구조와 맥락을 어떻게 남길 것인가"를 더 자주 생각하게 된다.

연차가 쌓이면서 자연스럽게 생긴 관심도 있다.
이제는 단순히 구현만 하는 사람이 아니라, 언젠가는 조직 관리나 기술 리딩에 더 가까운 역할도 해보고 싶다는 생각이 든다.

그런데 현실은 조금 다르다.
지금은 비IT 회사에서 1인 개발을 하고 있다.

- 서비스 구조를 정하는 사람도 나
- 코드를 짜는 사람도 나
- 배포하고 운영하는 사람도 나
- 장애가 나면 대응하는 사람도 나

당장은 이게 가능하다.
내가 다 알고 있으니까.

문제는 바로 거기에 있다.

> 내가 다 알고 있다는 건,
> 시스템이 문서가 아니라 사람 기억 위에 올라가 있다는 뜻이기도 하다.

지금은 혼자라서 버틴다.
하지만 미래까지 이 상태로 가는 건 맞지 않다는 생각이 들기 시작했다.

---

## 2. 조직 관리는 사람을 뽑는 순간 시작되는 게 아니었다

처음에는 "팀이 생기면 그때 문서를 정리하면 되지 않을까"라고 생각했다.
그런데 돌아보면 그건 순서가 반대였다.

사람을 뽑는 순간 조직이 생기는 게 아니라,
**내가 가진 지식을 남기기 시작하는 순간부터 조직의 기반이 생긴다.**

문서가 없으면 나중에 사람이 들어와도 이런 일이 반복될 가능성이 크다.

- 왜 이 기술을 골랐는지 설명이 안 된다
- 배포 순서가 사람 기억에만 있다
- 장애가 나면 누가 봐도 같은 대응을 하기가 어렵다
- 새 사람이 들어오면 코드보다 맥락 설명에 시간이 더 든다
- 몇 달 뒤의 나도 "왜 이렇게 했지?"를 다시 추적하게 된다

결국 조직 관리에 대한 관심은 사람 관리 이전에,
**지식과 의사결정을 어떻게 남길 것인가**의 문제로 먼저 오더라.

---

## 3. 이전 회사의 운영팀장, 개발팀장들에게서 본 것

이 생각을 하게 된 배경에는 예전 조직에서 봤던 리드들의 모습도 있다.

당시에는 그냥 "체계가 있네" 정도로 느꼈다.
그런데 지금 1인 개발을 하다 보니, 그 체계의 정체가 조금 더 선명하게 보인다.

이전 회사의 운영팀장, 개발팀장들은 코드만 보는 사람들이 아니었다.
그 사람들은 대체로 이런 것들을 관리하고 있었다.

- 왜 이렇게 운영하는지
- 문제가 나면 어디부터 봐야 하는지
- 배포할 때 어떤 순서를 따라야 하는지
- 누가 어떤 책임을 갖는지
- 새로운 사람이 들어왔을 때 무엇부터 이해해야 하는지

돌아보면 그건 결국 문서와 절차였다.

그들은 시스템을 사람 기억으로 운영하지 않으려고 했다.
장애 대응도, 배포도, 의사결정도 "아는 사람이 알아서"가 아니라, 가능한 한 남겨진 기준 위에서 움직이게 하려고 했다.

나는 지금 그 정도 구조 안에 있지 않다.
비IT 회사의 1인 개발자라는 환경에서는 오히려 반대로 가기 쉽다.

- 내가 다 아니까 그냥 내가 처리하고
- 문서 남길 시간에 기능 하나 더 만들고
- 운영 절차도 머릿속으로 외우고 넘어간다

당장은 이게 가장 빠르다.
하지만 미래 조직을 생각하면 이게 가장 비싼 방식일 수도 있다는 생각이 든다.

---

## 4. 지금 내 환경이 특히 더 위험한 이유

비IT 회사의 1인 개발은 문서화가 더 중요해질 수밖에 없다.

왜냐하면 이런 조건이 겹치기 쉽기 때문이다.

- 기술 의사결정을 설명할 상대가 당장 없다
- 그래서 결정이 자연스럽게 머릿속에만 남는다
- 운영 절차도 표준화되지 않고 개인 습관이 된다
- 나 말고는 시스템 전체를 이해하는 사람이 없다

Jira 같은 일정 관리 툴이라도 팀 단위로 굴러가고 있었다면, 적어도 어떤 일이 왜 들어왔고 어떤 순서로 처리됐는지의 흔적은 남았을 것이다.
그런데 지금은 그런 협업 도구의 흐름조차 약하다.
일정, 우선순위, 의사결정 맥락이 이슈 트래킹 시스템이 아니라 사람 머릿속과 메신저 대화에 흩어지기 쉽다.

문서화가 더 중요하게 느껴지는 이유도 여기 있다.
코드 바깥의 맥락이 남지 않기 때문이다.

이 구조는 흔히 말하는 bus factor가 1인 상태다.
내가 하루 자리를 비워도 당장 큰일이 날 수 있고,
몇 달 뒤의 나조차 과거의 결정을 다시 해석해야 할 수 있다.

즉, 문서화는 "팀이 크면 필요한 것"이 아니라,
오히려 **팀이 없을수록 더 빨리 필요한 것**일 수도 있다.

---

## 5. 문서도 다 같은 문서는 아니다

문서가 필요하다고 느끼기 시작하면 처음에는 막막하다.
"문서화를 해야겠다"는 생각은 드는데, 무엇부터 남겨야 할지가 잘 안 보인다.

정리해보니 문서는 대략 네 묶음으로 나눌 수 있었다.

### 1) 의사결정 문서

이건 "왜 이렇게 했는가"를 남기는 문서다.

| 약어/용어 | 풀네임 | 설명 |
|-----------|--------|------|
| ADR | Architecture Decision Record | 아키텍처 결정 기록 |
| RFC | Request for Comments | 제안과 피드백 중심 문서 |
| DR | Decision Record | 아키텍처 외 결정까지 포함하는 더 넓은 개념 |

### 2) 설계 / 명세 문서

이건 "무엇을 만들고, 어떻게 만들 것인가"를 남기는 문서다.

| 약어/용어 | 풀네임 | 설명 |
|-----------|--------|------|
| Design Doc / Tech Spec | - | 기능 구현 전 기술 설계서 |
| PRD | Product Requirements Document | 무엇을 왜 만드는가 |
| SRS | Software Requirements Specification | 더 정형화된 요구사항 명세 |
| ERD | Entity Relationship Diagram | DB 관계 구조 |
| API Spec | API Specification | API 명세 |

### 3) 운영 문서

이건 "문제가 나면 어떻게 대응할 것인가"를 남기는 문서다.

| 약어/용어 | 풀네임 | 설명 |
|-----------|--------|------|
| SOP | Standard Operating Procedure | 표준 운영 절차서 |
| Runbook | - | 장애 발생 시 단계별 대응 매뉴얼 |
| Postmortem | - | 장애 회고 문서 |
| RCA | Root Cause Analysis | 근본 원인 분석 |
| SLA / SLO / SLI | Service Level Agreement / Objective / Indicator | 서비스 수준 기준 |

### 4) 온보딩 / 가이드 문서

이건 "새 사람이 들어왔을 때 어디서부터 봐야 하는가"를 남기는 문서다.

| 약어/용어 | 풀네임 | 설명 |
|-----------|--------|------|
| README | - | 프로젝트 진입점 |
| Contributing Guide | - | 기여 규칙과 PR 규칙 |
| Changelog | - | 변경 이력 |

이렇게 나눠보니 막연했던 "문서화"가 조금 현실적인 작업으로 보이기 시작했다.

---

## 6. 그런데 모든 문서를 다 만들 수는 없다

여기서 중요한 건 하나다.

> 모든 문서를 다 만들겠다는 게 아니다.
> 지금 내 상황에서 ROI가 큰 문서부터 남기겠다는 것이다.

1인 개발자가 처음부터 모든 문서 체계를 다 가져가면 오히려 문서가 부담이 된다.
유지되지 않는 문서는 없는 것보다 더 나쁘다.

그래서 우선순위가 필요했다.

### 지금 바로 남길 가치가 큰 것

- `ADR`
- `README`
- `ERD`
- `API Spec`
- `Runbook`

이 다섯 개는 지금 상황에서도 바로 효용이 있다.

`ADR`은 몇 달 뒤의 나를 위해 필요하다.
"왜 Redis를 안 썼는지", "왜 tenant_id 대신 별도 B2B 테이블을 뒀는지" 같은 판단은 시간이 지나면 기억보다 문서가 더 정확하다.

`README`는 프로젝트의 진입점이다.
빌드, 실행, 모듈 구조, 환경 변수 정도만 정리돼 있어도 새 사람이 들어왔을 때 설명 비용이 크게 줄어든다.

`ERD`와 `API Spec`은 말 그대로 구조를 보여준다.
설명보다 빠르다.

`Runbook`은 운영 문서 중 가장 실용적이다.
장애가 났을 때 "어디부터 확인할지"만 정리돼 있어도 대응 품질이 달라진다.

### 팀이 생기면 더 강해지는 것

- `RFC`
- `Contributing Guide`
- `SOP`
- `Postmortem`

이 문서들은 혼자 있을 때보다, 여러 사람이 협업할 때 훨씬 힘을 발휘한다.
하지만 그렇다고 팀이 생긴 뒤에 처음 고민할 주제는 아니다.
지금부터 최소한의 틀은 잡아둘 수 있다.

### 아직은 과한 것

- 형식이 너무 무거운 `PRD`
- 과도하게 정형화된 `SRS`
- 운영 조직 없이 선언만 있는 `SLA / SLO / SLI`

이런 것들은 필요 없다고 보진 않는다.
다만 지금 내 환경에서 바로 ROI가 높다고 보긴 어렵다.

문서도 결국 기능과 마찬가지로,
**지금 단계에서 감당 가능한 수준으로 도입해야 한다.**

---

## 7. 지금 특히 더 남기고 싶은 건 ADR이다

여러 문서 중에서 지금 가장 먼저 의식하게 되는 건 ADR이다.

이유는 단순하다.
내 블로그에 쓰고 있는 많은 글들이 사실상 ADR의 재료에 가깝기 때문이다.

예를 들면 이런 것들이다.

- 왜 Facade를 Application 레이어에 뒀는가
- 왜 JSONB 대신 EAV를 골랐는가
- 왜 풀 멀티테넌시 대신 B2B 오버레이를 택했는가
- 왜 결과지 구조를 Transfer Layer로 분리했는가

이건 전부 "무엇을 구현했는가"보다
**왜 그 선택을 했는가**에 가까운 기록이다.

블로그 글은 외부 공유용으로 쓰기 좋다.
하지만 내부 시스템 관점에서는 더 짧고, 더 직접적인 결정 문서도 필요하다.

예를 들면 이런 형태다.

```text
ADR-007: B2B 대상자는 기존 member 테이블에 통합하지 않는다

배경:
- 현재 B2C 경로는 Voucher -> Member 전제를 강하게 갖고 있음
- B2B는 회원보다 이용권/검사 흐름이 우선임

결정:
- B2B는 b2b_voucher, b2b_voucher_kit 별도 테이블 사용

대안:
- 기존 voucher/member에 tenant_id 추가

선택 이유:
- 기존 B2C 코드 영향 최소화
- 마이그레이션 비용 절감

후속 영향:
- 리포트 조회 경로가 별도로 필요함
```

이 정도만 남겨도,
몇 달 뒤 "왜 이걸 이렇게 했지?"를 다시 파헤치는 시간을 꽤 줄일 수 있다.

---

## 8. 문서화는 미래의 팀뿐 아니라 미래의 나를 위한 것이기도 하다

사실 이 글을 쓰게 된 이유의 절반은 미래의 팀이고,
나머지 절반은 미래의 나다.

혼자 개발할 때 문서화가 가장 먼저 무너지는 이유는 간단하다.
"내가 다 아는데 굳이?"라는 생각이 너무 자연스럽기 때문이다.

그런데 몇 달만 지나도 상황이 달라진다.

- 왜 그때 그 구조를 골랐는지 기억이 흐려진다
- 결정 당시 제약 조건이 뭐였는지 잊어버린다
- 같은 문제를 처음부터 다시 검토하게 된다

결국 문서화는 협업 효율을 위한 것만이 아니다.
**의사결정의 재학습 비용을 줄이는 장치**이기도 하다.

---

## 9. 정리

지금 나는 비IT 회사의 1인 개발자다.
아직 내 아래에 팀이 있는 것도 아니고, 운영 조직이 체계적으로 붙어 있는 것도 아니다.

그렇다고 미래까지 1인 개발자일 거라고 가정하고 시스템을 만들 수는 없다.

오히려 이런 환경일수록 더 빨리 느끼게 된다.

- 코드만으로는 조직이 생기지 않는다
- 사람을 뽑는다고 바로 팀이 되는 것도 아니다
- 지식과 의사결정이 남아 있어야 비로소 확장이 가능하다

그래서 지금 내 결론은 단순하다.

> 팀이 생긴 뒤에 문서를 만들겠다는 건 늦다.
> 사람보다 먼저 남겨야 하는 건 지식이고,
> 그 지식을 가장 현실적으로 남기는 방법이 문서다.

모든 문서를 다 만들겠다는 건 아니다.
하지만 적어도 `README`, `ADR`, `ERD`, `API Spec`, `Runbook` 정도는
지금부터 차근차근 쌓아둘 필요가 있다고 본다.

조직 관리를 해보고 싶다는 생각은
결국 사람을 관리하고 싶다는 뜻만은 아닌 것 같다.
이제는 시스템이 사람 기억이 아니라 남겨진 기준 위에서 돌아가게 만들고 싶다는 뜻에 더 가깝다.
